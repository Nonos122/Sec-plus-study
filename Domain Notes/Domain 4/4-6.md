# 4.6 Notes

## Provisioning User Accounts

The process of creating and granting a new user the access they need, ideally following least privilege from day one.

### Active Directory

Microsoft's directory service used to centrally manage users, computers, and group policy across a Windows domain; provisioning here typically means creating the user object, assigning group memberships, and applying the appropriate policies.

### New User Accounts

Should be created with only the access required for the person's role, tied to a documented approval/request process, and ideally sourced from an authoritative system (like HR) to prevent accounts from being created without proper justification.

### Kerberos

A ticket-based authentication protocol (used heavily by Active Directory) where a trusted Key Distribution Center (KDC) issues time-limited tickets, allowing a user to authenticate once and access multiple resources without re-entering credentials each time.

### Linux

User provisioning on Linux typically involves creating accounts (`useradd`), assigning them to appropriate groups, and applying permissions through the standard Linux permission model or tools like SELinux for more granular mandatory access control.

## Deprovisioning User Accounts

Promptly disabling or removing a user's access when they leave the organization or change roles, so old credentials can't be used for unauthorized access. Deprovisioning should be timely and complete — covering not just the primary account, but any linked accounts, access keys, or group memberships tied to that person.

## Permission Assignments and Implications

Access decisions can be informed by more than just a static role — additional context can be factored in:

1. Location — restricting or adjusting access based on the physical or network location a request comes from.
2. Time — allowing access only during expected hours (e.g., business hours), flagging or blocking requests outside that window.
3. Device — considering whether the request comes from a known, managed, and compliant device.
4. Network — considering whether the connection comes from a trusted internal network vs. an untrusted external one.
5. Biometric — using a physical characteristic as an additional condition for granting access to especially sensitive resources.

*Implication:* the more of these factors that are combined, the more an access control model starts to resemble Attribute-Based Access Control (ABAC) rather than a simple role-based system.

## Identity Proofing

The process of verifying that a person requesting an account or access truly is who they claim to be, before that identity is trusted — often involving government ID verification, knowledge-based questions, or in-person verification, especially for high-privilege accounts.

## Federation

Allowing a user's identity from one trusted domain/organization (the identity provider) to be used to access resources in another domain/organization (the service provider), without needing to create and manage a separate account in each system.

## Single Sign-On (SSO)

Allows a user to authenticate once and gain access to multiple, separate applications/systems without logging in again for each one.

1. Kerberos authentication — commonly used for SSO within a single Windows Active Directory domain, using tickets issued by a KDC.
2. OAuth — a token-based framework mainly used for delegated authorization, commonly seen in "Sign in with Google/Microsoft" style access, letting an app access resources on a user's behalf without handling their password directly.
3. SAML (Security Assertion Markup Language) — an XML-based standard for exchanging authentication and authorization data between an identity provider and a service provider, commonly used for enterprise/web SSO.

## Interoperability

The ability of different identity systems, directories, and applications (potentially from different vendors) to work together and exchange authentication/authorization information reliably — essential for federation and SSO to function across an organization's mixed technology stack.

## Attestation

The process of providing verifiable proof of an identity, device, or compliance status, often relying on:

1. Certificates — digital certificates proving identity, signed by a trusted CA.
2. Tokens — physical or digital tokens that prove possession of a credential.
3. Federation — trusting another organization's identity provider's attestation of a user's identity.
4. Microsoft Active Directory — acting as a central authority that can attest to a user's identity and group memberships within a domain.

## Access Controls

### MAC (Mandatory Access Control)

Access is controlled centrally using security labels/classifications (e.g., Confidential, Secret, Top Secret); users cannot change permissions themselves, since access decisions are enforced by the system based on policy, not user discretion. Commonly used in government/military environments.

*Note:* Owner, steward, custodian, and security administrator are data governance roles (who is accountable for data and its protection) rather than part of the MAC model itself — worth knowing separately: the owner is accountable for classification, the steward manages data quality/policy day-to-day, the custodian handles technical storage/security, and the security administrator manages the technical controls enforcing access.

### RBAC (Role-Based Access Control)

Access is granted based on a user's assigned role within the organization (e.g., "HR Manager" or "Help Desk Technician"), simplifying management by grouping permissions around job functions rather than individuals.

### ABAC (Attribute-Based Access Control)

Access decisions consider multiple attributes together (user role, resource sensitivity, time, location, device) for fine-grained, context-aware control, going beyond what RBAC alone can express.

### DAC (Discretionary Access Control)

The owner of a resource decides who else can access it and what they can do — flexible, but less centrally controlled and more prone to inconsistent permissions over time.

### Time-of-Day Restrictions

Limiting when a user or account is allowed to log in or access resources (e.g., only during business hours), reducing the window of opportunity for misuse of stolen credentials outside those hours.

### Least Privilege

Users, applications, and systems should be granted only the minimum access necessary to perform their function — nothing more.

## Multifactor Authentication (MFA)

Requiring two or more independent forms of verification before granting access, combining factors from different categories.

### Biometric

Physical characteristics used to verify identity:

- Fingerprint
- Retina — scans the blood vessel pattern at the back of the eye.
- Iris — scans the colored ring pattern around the pupil.
- Voice
- Facial
- Vein pattern
- Gait — a person's unique walking pattern.

**Biometric error rates:**

- FAR (False Acceptance Rate) — the rate at which an unauthorized person is incorrectly accepted as a match.
- FRR (False Rejection Rate) — the rate at which an authorized person is incorrectly rejected.
- CER (Crossover Error Rate) — the point where FAR and FRR are equal; a lower CER indicates a more accurate biometric system overall.

### Hard Authentication

Physical devices used to authenticate:

- Smart card
- Fobs (hardware tokens generating one-time codes)
- Security keys (e.g., FIDO2/YubiKey devices)
- SSH keys — a public/private key pair used to authenticate without a password, commonly for server administration.

### Soft Authentication

Software-based authentication methods:

- OTP (One-Time Password) — generated by an authenticator app rather than a physical device.
- Biometric authentication — using a device's built-in sensor (like a phone's fingerprint reader) rather than a dedicated hardware token.
- KBA (Knowledge-Based Authentication) — verifying identity through answers to personal security questions.

### Factors of Authentication

- Something you know — a password or PIN.
- Something you have — a token, smart card, or phone.
- Something you are — a biometric trait.
- Something you do — a behavioral pattern, like typing rhythm or gait.
- Somewhere you are — your location, verified through GPS or network information.
- Single factor — using only one of the above categories.
- Two factor — using exactly two different categories together.
- Multifactor — using two or more different categories together (two-factor is a subset of multifactor).

### Tokens

- RSA SecurID — a hardware/software token generating time-based one-time passcodes for authentication.
- Google Authenticator — a software-based OTP generator app using the TOTP standard.

## Password Concepts

- Password length — longer passwords are exponentially harder to brute-force; modern guidance favors length over complex but short passwords.
- Password complexity — requiring a mix of character types (upper/lowercase, numbers, symbols) to make guessing harder.
- Password reuse — using the same password across multiple accounts/systems, which should be avoided since one breach can then compromise many accounts.
- Password expiry — requiring passwords to be changed after a set period (note: modern guidance, like NIST, has moved away from frequent forced expiration in favor of length/MFA, since forced expiry often leads to weaker, predictable password patterns).
- Password age
  - Minimum age — a required waiting period before a password can be changed again, preventing users from quickly cycling back to an old password.
  - Maximum age — the longest period a password can be used before it must be changed.
- Account lockout — automatically locking an account after a set number of failed login attempts, to slow down brute-force and password-spraying attacks.

## Password Managers

Applications that securely generate, store, and autofill unique, complex passwords for each account, reducing password reuse and the burden of memorizing many strong passwords.

## Passwordless

Authentication methods that eliminate the traditional password entirely, relying instead on possession (security keys, device-based authentication) and/or biometrics (e.g., FIDO2/WebAuthn), reducing risk from phishing, credential stuffing, and password reuse.

## PAM (Privileged Access Management)

A set of tools and practices focused specifically on securing, controlling, and monitoring accounts with elevated/administrative privileges, since these accounts pose the highest risk if compromised.

## PAM Tools

- JIT (Just-in-Time) permissions — granting elevated access only for the specific time window it's needed, rather than leaving privileged access active permanently.
- Password vaulting — storing privileged credentials in a secure, centrally managed vault, checking them out only when needed and rotating them automatically afterward.
- Ephemeral credentials — temporary credentials generated for a single session or task that automatically expire, so there's no long-lived privileged credential to steal.
