# 4.3 Notes

## Identification Methods

### Vulnerability Scans

1. Non-credentialed scan — scans a target without login credentials, seeing only what's visible externally, like an outside attacker would.
2. Credentialed scan — scans a target using valid credentials, providing a deeper, more accurate view of vulnerabilities from inside the system.

**Benefits:**

1. Identifying weak points — surfaces known vulnerabilities before an attacker can exploit them.
2. Prioritizing remediation — helps rank which vulnerabilities to fix first based on severity and exposure.
3. Continuous monitoring — regular scanning keeps the organization aware of new vulnerabilities as they emerge.

### Security Content Automation Protocol (SCAP)

1. OVAL (Open Vulnerability and Assessment Language) — a standardized format for describing system configuration, vulnerability, and patch state so tools can check systems consistently.
2. XCCDF (Extensible Configuration Checklist Description Format) — a standardized format for writing security checklists and benchmarks that can be processed automatically.

### Application Security

1. Static analysis — examining source code without executing it, to catch vulnerabilities early in development.
2. Dynamic analysis — testing a running application to observe its actual behavior and uncover vulnerabilities that only appear at runtime.
3. Web application scanners — automated tools that probe a live web application for common vulnerabilities like SQL injection and XSS.

### Package Monitoring

Continuously tracking the third-party libraries and dependencies used in an application to identify known vulnerabilities (e.g., a CVE affecting an open-source package) as soon as they're disclosed, so they can be patched or replaced.

### Threat Feeds

1. Security vendors — commercial security companies publishing intelligence on current threats and campaigns.
2. Government agencies — e.g., CISA (Cybersecurity and Infrastructure Security Agency), which publishes advisories and alerts on active threats.
3. OSINT (Open-Source Intelligence) — publicly available information gathered from sources like news, forums, and social media.
4. Commercial threat intelligence providers — paid services offering curated, in-depth threat intelligence feeds.

### OSINT Tools

Tools used to gather open-source intelligence, such as Shodan (searching internet-connected devices), theHarvester (collecting emails/subdomains), Maltego (mapping relationships between entities), and search engines/social media for reconnaissance.

### Proprietary/Third-Party

1. STIX (Structured Threat Information eXpression) — a standardized language for describing and sharing structured threat intelligence data.
2. SHODAN — a search engine that indexes internet-connected devices, often used to find exposed systems, open ports, and misconfigured devices.

### Information Sharing Organizations

1. IOCs (Indicators of Compromise) — evidence (file hashes, IPs, domains) that a system has been compromised.
2. TTPs (Tactics, Techniques, and Procedures) — the behavioral patterns describing how a threat actor operates.
3. Incident data — details from past incidents shared to help others recognize and prepare for similar attacks.
4. CTA (Cyber Threat Alliance) — an organization where member companies share threat intelligence to improve collective defense.
5. AIS (Automated Indicator Sharing) — a CISA program that enables real-time sharing of cyber threat indicators between the government and private sector.
6. FIRST (Forum of Incident Response and Security Teams) — a global organization connecting incident response teams to collaborate and share best practices.
7. ISACs (Information Sharing and Analysis Centers) — sector-specific organizations (finance, healthcare, energy, etc.) that share threat intelligence relevant to their industry.
8. MS-ISAC (Multi-State Information Sharing and Analysis Center) — an ISAC focused on state, local, tribal, and territorial government entities in the U.S.

### The Dark Web

A portion of the internet not indexed by standard search engines, accessible only through specific software (like Tor). Security teams monitor it to detect leaked credentials, stolen data for sale, or discussions of planned attacks against the organization.

### Pen Testing

1. Known environment (white box) — testers are given full knowledge of the target's systems, architecture, and source code ahead of time.
2. Partially known environment (gray box) — testers are given some information (e.g., network diagrams or limited credentials) but not full access.
3. Unknown environment (black box) — testers are given no prior knowledge, simulating a real external attacker's perspective.

### Responsible Disclosure Program

A formal process that gives security researchers a clear, legal way to report vulnerabilities they discover to an organization, allowing time for a fix before the details are made public.

### Bug Bounty

A program that pays external researchers a reward for finding and responsibly reporting vulnerabilities, incentivizing a wider pool of people to help find flaws before attackers do.

### System/Process Audit

A formal review of systems, configurations, and processes to verify they meet security policy, best practices, and compliance requirements, often performed by internal audit teams or external assessors.

## Analysis

### Confirmation

Verifying that a reported or scanned vulnerability is real and exploitable (rather than a false positive) before spending resources on remediation.

### Prioritization

1. CVE (Common Vulnerabilities and Exposures) — a publicly cataloged, uniquely identified entry describing a specific known vulnerability.
2. CVSS (Common Vulnerability Scoring System) — a standardized 0-10 score reflecting a vulnerability's severity, used to help prioritize remediation.

### Vulnerability Classification

Categorizing a vulnerability by type (e.g., injection flaw, misconfiguration, missing patch) and severity, to help route it to the right team and determine urgency.

### Exposure Factor

The percentage of an asset's value that would be lost if a specific vulnerability were successfully exploited; used as part of calculating potential impact (feeds into Single Loss Expectancy).

### Environmental Variable

Factors specific to an organization's own environment (network placement, compensating controls already in place, exposure to the internet) that adjust how severe a vulnerability actually is for that organization, beyond its generic CVSS score.

### Industry/Organizational Impact

How a vulnerability's exploitation would specifically affect the organization's operations, reputation, finances, or regulatory standing — since the same vulnerability can matter more or less depending on the industry and business context.

### Risk Tolerance

The level of risk an organization is willing to accept before taking action, which influences how quickly and aggressively vulnerabilities get remediated versus accepted.

## Vulnerability Response and Remediation

### Patching

Applying vendor-released updates to fix the underlying flaw a vulnerability is based on — the most direct form of remediation when available.

### Insurance

Transferring some of the financial risk of a vulnerability being exploited to a third party (a cyber insurance policy), used when the risk can't be fully mitigated through technical controls alone.

### Segmentation

Isolating a vulnerable system from the rest of the network as a compensating control, limiting what an attacker could reach if the vulnerability were exploited, when patching isn't immediately possible.

## Validation of Remediation

### Rescanning

1. Validation of effectiveness — confirming the vulnerability was actually fixed and no longer detected.
2. Identifying new issues — checking that the fix didn't introduce a new vulnerability or break something else.
3. Compliance verification — confirming the system now meets required security/compliance standards.

### Audit

1. Documentation review — checking that remediation was properly recorded.
2. Process adherence — confirming the fix followed the organization's approved change management process.
3. Compliance check — verifying the remediation satisfies relevant regulatory or policy requirements.

### Verification

1. Continuous monitoring — ongoing observation to ensure the fixed vulnerability doesn't reappear (e.g., through configuration drift).
2. Periodic assessments — scheduled reviews to confirm systems remain in a secure, compliant state over time.
3. Adaptive responses — adjusting security controls and processes based on what's learned from the remediation and verification process.

## Reporting

1. Vulnerability overview — a summary of what vulnerabilities were found and their general nature.
2. CVSS scores — the severity ratings for each identified vulnerability.
3. Remediation process — what was done (or is planned) to fix each vulnerability.
4. Risk reduction — how much the organization's overall risk was lowered as a result of remediation.
5. Recommendations — suggested next steps, including any vulnerabilities not yet remediated and why.
