# 4.5 Notes — Very Important for PBQs

## Firewalls

| Firewall Type | Mission (What It Does) | Typical Use Case |
|---|---|---|
| Host-based | Runs on a single device/endpoint, filtering traffic to and from that specific host only. | Protecting an individual laptop or server, especially one that travels outside the corporate network (e.g., a remote worker's laptop). |
| Network-based | A dedicated appliance/device filtering traffic for an entire network segment. | Protecting the perimeter of a company network, sitting between the internet and internal systems. |
| Stateful | Tracks the full context of a connection (its "state") and automatically allows return traffic tied to an already-established session. | General-purpose perimeter firewalls where you want to allow a response to traffic you already permitted out, without writing a rule for every possible reply. |
| Stateless | Evaluates every packet individually against static rules, with no awareness of connection state. | High-throughput environments needing simple, fast filtering (e.g., basic ACLs on a router) where deep connection tracking isn't required. |
| WAF (Web Application Firewall) | Inspects and filters HTTP/HTTPS traffic at the application layer, understanding web-specific attacks. | Sitting in front of a web application to block SQL injection, XSS, and other application-layer attacks. |
| UTM (Unified Threat Management) | Combines multiple security functions (firewall, antivirus, IPS, content filtering, VPN) into a single appliance. | Small/medium businesses that want broad protection from one device rather than managing many separate tools. |
| NGFW (Next-Generation Firewall) | Adds deep packet inspection, application awareness, user/identity-based rules, and integrated IPS on top of traditional firewall filtering. | Enterprise environments needing granular control over specific applications and users, not just ports/IPs. |

### Rules

1. Inbound and outbound rules — inbound rules control traffic coming into a network/host; outbound rules control traffic leaving it. Both need to be defined, since blocking only inbound traffic still allows a compromised host to send data out (e.g., to a C2 server).
2. Explicit allow and explicit deny — rules that specifically state whether matching traffic is permitted or blocked (rather than relying only on a default/implicit rule at the end of the list). Rules are typically built using:
   - Port numbers — which service the traffic is destined for (e.g., 443 for HTTPS).
   - Protocol types — TCP, UDP, ICMP, etc.
   - IP addresses — the source and/or destination address(es) the rule applies to.

### Access Control List (ACL)

A set of ordered rules configured on a network device (router, switch, firewall) that explicitly permit or deny traffic based on criteria like source/destination IP, port, and protocol. ACLs are processed top-down, stopping at the first matching rule, with an implicit "deny all" applied at the end for anything that doesn't match. Rule order matters — a broad rule placed above a more specific one can unintentionally block or allow traffic you didn't intend.

## Ports and Protocols

### TCP (Transmission Control Protocol)

A connection-oriented protocol that establishes a reliable session before sending data, using a three-way handshake:

1. SYN — the client sends a synchronize packet to initiate the connection.
2. SYN-ACK — the server responds, acknowledging the request and synchronizing its own sequence number.
3. ACK — the client acknowledges the server's response, completing the handshake; the connection is now established.

Once established, TCP manages the actual data exchange through:

1. Data transmission — sending data in ordered segments.
2. Acknowledgment — confirming receipt of each segment so the sender knows it arrived.
3. Error handling — detecting lost or corrupted segments and retransmitting them, ensuring reliable delivery.

### UDP (User Datagram Protocol)

A connectionless protocol that sends data without establishing a session or guaranteeing delivery — faster than TCP but less reliable, used where speed matters more than guaranteed delivery (e.g., DNS lookups, video streaming, VoIP).

### ICMP (Internet Control Message Protocol)

Used for network diagnostics and error reporting rather than carrying application data — e.g., `ping` (echo request/reply) and `traceroute` rely on ICMP to test connectivity and map network paths.

### ANY

A rule/protocol wildcard meaning "match traffic regardless of protocol/port/source." Firewall rules using "any" are much broader and riskier, since they don't restrict traffic by any specific criteria — generally avoided except in very deliberate, narrow circumstances.

### Port Number Ranges

| Range | Category | Description | Examples |
|---|---|---|---|
| 0–1023 | Well-known ports | Reserved for common, standardized services; assigned by IANA. | 21 (FTP), 22 (SSH), 25 (SMTP), 53 (DNS), 80 (HTTP), 443 (HTTPS) |
| 1024–49151 | Registered ports | Registered with IANA for specific applications/services, but not as tightly controlled as well-known ports. | 1433 (Microsoft SQL Server), 3306 (MySQL), 3389 (RDP), 8080 (alternate HTTP) |
| 49152–65535 | Dynamic/private ports | Not registered to any specific service; used temporarily by client applications for outbound connections (ephemeral ports). | Randomly assigned source ports for outgoing client connections |

## Zones

1. WAN (Wide Area Network) — the broader external network (typically the internet) that connects the organization to the outside world.
2. LAN (Local Area Network) — the internal, trusted network where an organization's users and systems operate.
3. Screened subnet (DMZ) — a buffer zone between the WAN and LAN that hosts public-facing services, isolated so a compromise there doesn't directly expose the internal LAN.
4. Untrusted zone — any network segment (often the WAN/internet) treated as inherently unsafe, with no implicit trust granted to traffic from it.
5. Boundary layer — the point where two zones meet and traffic is inspected/filtered (typically enforced by a firewall) as it crosses from one zone to another.
6. Trusted zone — an internal network segment where systems and users are given a higher baseline level of trust (though Zero Trust principles still apply verification regardless of zone).

## IDS/IPS

### Trends in IDS/IPS

1. Machine learning and AI integration — using ML models to detect novel or subtle attack patterns that static signatures would miss.
2. Cloud-centric security — IDS/IPS capabilities delivered and managed through cloud platforms rather than only on-prem appliances.
3. Zero Trust architecture — IDS/IPS increasingly integrated into a broader Zero Trust model, continuously verifying traffic rather than just guarding the perimeter.
4. IoT and OT protection — extending detection/prevention capabilities to cover the growing number of IoT and operational technology devices, which often can't run traditional security agents.

### IDS/IPS Signatures

1. Signature-based detection — matches traffic against a database of known attack patterns; effective against known threats but blind to new ones.
2. Anomaly/heuristic-based detection — establishes a baseline of "normal" behavior and flags deviations from it, catching novel attacks at the cost of more false positives.
3. Real-time updates — signature databases are updated continuously so the system can recognize newly discovered threats quickly.
4. Custom signatures — organizations can write their own detection rules tailored to threats specific to their environment.

## Web Filtering

1. Agent-based filtering — software installed directly on the endpoint to filter web traffic, effective even when the device is off the corporate network.
2. Centralized proxy filtering — all web traffic is routed through a central proxy server that applies filtering rules before traffic reaches the internet.
3. URL scanning — checking requested URLs against known-malicious or restricted lists in real time.
4. Content categorization — classifying websites by category (e.g., gambling, social media, malware) so policies can block entire categories at once.
5. Block rules — explicit rules defining specific sites, categories, or content types to block.
6. Reputation-based filtering — blocking sites based on a dynamically updated reputation score reflecting how trustworthy or risky that site has historically been.

## Operating System Security

1. Keep your system updated — apply OS and application patches promptly to close known vulnerabilities.
2. UAC (User Account Control) — prompts for elevated permission before allowing actions that require administrative rights, reducing the risk of unauthorized changes.
3. Minimize attack surface — disable unnecessary services, ports, and features that aren't needed.
4. Implement strong authentication — enforce strong passwords and multi-factor authentication.
5. Employ access controls — apply least privilege so users/processes only have the access they actually need.
6. Enable firewall protection — use host-based firewalls to filter traffic to and from the system.
7. Encrypt data — protect data at rest (e.g., full-disk encryption) and in transit.
8. Monitor and log activities — maintain logs of system activity to detect and investigate suspicious behavior.
9. Patch management — follow a structured, tested process for applying updates consistently.
10. Educate users — train users on secure practices, since human error is a major risk factor regardless of technical controls.
11. Back up your data — maintain regular, tested backups to recover from data loss or ransomware.
12. Disaster recovery plan — have a documented plan for restoring systems and operations after a major disruption.

## Group Policy

A Windows feature that allows administrators to centrally define and enforce configuration settings (security policies, software restrictions, password requirements) across many computers and users within an Active Directory domain, ensuring consistent baseline security without manually configuring each machine.

## SELinux

Security-Enhanced Linux — a Linux kernel security module that enforces Mandatory Access Control (MAC), restricting what processes and users can do based on defined security policies, beyond the traditional discretionary Linux permission model. It significantly limits the damage a compromised process can do, even if it's running with elevated privileges.

## Implementation of Secure Protocols

1. Protocol selection — choosing a secure version of a protocol (e.g., HTTPS instead of HTTP, SSH instead of Telnet) whenever one is available.
2. Port selection — ensuring the correct, expected port is used for the secure protocol, and that insecure alternatives aren't left open alongside it.
3. Transport method — considering how the protocol carries data (e.g., encrypted tunnel vs. plaintext), ensuring encryption is actually applied end-to-end rather than just on part of the path.

## DNS Filtering

1. Blocks access to malicious sites — prevents users from resolving domains known to host malware, phishing, or C2 infrastructure.
2. Content filtering — blocks categories of sites (e.g., gambling, adult content) at the DNS level, before a connection is even made.
3. Enhancing privacy — can prevent DNS queries from leaking browsing habits to untrusted resolvers.
4. Security reinforcement — adds a layer of protection that works even if a user clicks a malicious link, since the domain never resolves.

## Email Security

1. S/MIME — encrypts and digitally signs individual email messages end-to-end using certificates.
2. PGP (Pretty Good Privacy) — provides encryption and digital signatures for email, similar in purpose to S/MIME but using a different trust model (web of trust vs. PKI/CA).
3. DMARC (Domain-based Message Authentication, Reporting & Conformance) — tells receiving mail servers what to do with messages that fail SPF/DKIM checks (quarantine, reject) and provides reporting back to the domain owner.
4. DKIM (DomainKeys Identified Mail) — adds a digital signature to outgoing email, allowing the receiving server to verify the message wasn't altered and genuinely came from the claimed domain.
5. SPF (Sender Policy Framework) — a DNS record listing which mail servers are authorized to send email on behalf of a domain, helping detect spoofed senders.
6. Gateway — a secure email gateway inspects incoming/outgoing mail for spam, phishing, malware, and policy violations before it reaches the user's inbox.

## File Integrity Monitoring (FIM)

A control that tracks changes to critical files, configurations, and system components, comparing their current state against a known-good baseline and alerting when unexpected or unauthorized modifications occur — useful for detecting tampering, malware, or unauthorized changes that other tools might miss.

## DLP (Data Loss Prevention)

A set of tools and policies designed to detect and prevent sensitive data from leaving the organization improperly — inspecting content in emails, file transfers, removable media, and cloud uploads, then blocking, flagging, or encrypting data that violates policy.

## NAC (Network Access Control)

Enforces security policy before and during a device's connection to the network, ensuring only compliant, authorized devices gain access.

1. Agents
   - Permanent agents — installed persistently on a device to continuously check compliance (patch level, antivirus status) each time it connects.
   - Dissolvable agents — temporary agents that run only during the connection session (e.g., for guest or BYOD devices) and remove themselves afterward.
2. Health authority — the component that evaluates a device's compliance status against defined policy before granting access.
3. Remediation server — directs non-compliant devices to a restricted area where they can be updated or fixed (e.g., patched, given updated antivirus definitions) before being granted full network access.

## Endpoint Detection and Response (EDR) and Extended Detection and Response (XDR)

**EDR:**

1. Data collection — continuously gathers detailed activity data from endpoints (processes, file changes, network connections).
2. Detection — analyzes that data to identify malicious or suspicious behavior.
3. Alerting — notifies security teams when a potential threat is detected.
4. Response — allows analysts (or automated rules) to take action, such as isolating an endpoint or killing a malicious process.

**XDR** (extends EDR's concept across more data sources):

1. Data integration — combines data from endpoints, network, cloud, and email into a single unified view, rather than looking at endpoints alone.
2. Advanced analytics — correlates data across all these sources to detect complex, multi-stage attacks that a single data source might miss.
3. Automation and orchestration — automatically triggers coordinated responses across multiple systems, not just the endpoint.
4. Scalability — designed to handle the larger volume and variety of data that comes from integrating many sources at once.

## User Behavior Analytics (UBA/UEBA)

Establishes a baseline of normal behavior for each user (typical login times, locations, resources accessed) and flags deviations that could indicate a compromised account, insider threat, or credential misuse — even when the activity uses valid, legitimate credentials and wouldn't otherwise trigger a signature-based alert.
