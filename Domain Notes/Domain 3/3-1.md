# 3.1 Notes

## Securing the Network

Applying controls — segmentation, firewalls, encryption in transit, monitoring (SIEM/IDS/IPS) — to protect data as it moves between systems and to limit how far an attacker can move if they get in.

## Securing Servers

Hardening the systems that host critical services: disabling unnecessary services, applying patches promptly, enforcing least privilege for accounts and processes, and using host-based firewalls/EDR — since a compromised server often has broad access to sensitive data.

## Securing Hosts

Endpoint-level protections applied to individual devices (workstations, laptops, mobile devices): disk encryption, host-based firewalls, EDR, application allow lists, and keeping the OS and software patched.

## Architecture & Infrastructure

The overall design of how systems, networks, and services are built and interconnected. Security has to be considered at the design phase ("secure by design") rather than bolted on afterward, since architecture decisions (cloud vs. on-prem, centralized vs. decentralized, segmentation approach) shape what's even possible to secure later.

## Cloud Computing

1. Public cloud — infrastructure owned and operated by a third-party provider, shared across many customers (e.g., AWS, Azure, GCP general offerings).
2. Private cloud — infrastructure dedicated to a single organization, either self-hosted or hosted by a provider exclusively for that organization.
3. Community cloud — infrastructure shared by several organizations with common requirements (e.g., regulatory needs), such as a group of government agencies.
4. Hybrid cloud — a mix of two or more cloud types (or cloud + on-premises) that work together, allowing data/workloads to move between them.
5. IaaS (Infrastructure as a Service) — the provider supplies raw compute, storage, and networking; the customer manages the OS, applications, and data.
6. PaaS (Platform as a Service) — the provider manages the OS and runtime; the customer manages applications and data.
7. SECaaS (Security as a Service) — security functions (e.g., email filtering, threat intelligence, SIEM) delivered as a cloud-based subscription service.
8. XaaS, NaaS, DaaS — "Anything as a Service" is the general category for cloud-delivered offerings; NaaS (Network as a Service) delivers networking capabilities (routing, load balancing, SD-WAN) as a service; DaaS (Desktop as a Service) delivers virtual desktops hosted and managed in the cloud.

## Responsibility Matrix

An example of how security responsibility shifts between customer and provider across service models:

| Layer | On-Premises | IaaS | PaaS | SaaS |
|---|---|---|---|---|
| Data & access management | Customer | Customer | Customer | Customer |
| Applications | Customer | Customer | Customer | Provider |
| Runtime/OS | Customer | Customer | Provider | Provider |
| Virtualization | Customer | Provider | Provider | Provider |
| Physical infrastructure (servers, storage, networking) | Customer | Provider | Provider | Provider |

The general rule: the more managed the service, the more security responsibility shifts to the provider — but the customer is always responsible for their own data and access controls.

## Hybrid Considerations

1. Data breaches — data moving between on-prem and cloud environments creates more potential points where it could be exposed or intercepted.
2. Security vulnerabilities — inconsistent controls or configurations between the two environments can leave gaps that don't exist in either environment alone.
3. Compliance challenges — data subject to regulations (e.g., PCI DSS, HIPAA) can be harder to track and control once it spans multiple environments.
4. Operational disruption — added complexity in connecting and synchronizing environments increases the risk of misconfiguration, outages, or failed failovers.

## Infrastructure as Code (IaC)

Managing and provisioning infrastructure (servers, networks, configurations) through machine-readable definition files instead of manual setup.

1. Consistency and reproducibility — the same code produces the same environment every time, eliminating configuration drift and manual errors.
2. Version control and collaboration — infrastructure definitions can be tracked in a version control system like Git, giving a full history of changes and enabling team collaboration.
3. Providers and tools — common tools include Terraform, Ansible, AWS CloudFormation, and Azure Resource Manager (ARM) templates.

## Serverless

A cloud model (often called Function as a Service, or FaaS) where the provider fully manages the underlying servers, and code runs only in response to specific triggers/events. The customer never manages the OS or runtime — but is still responsible for securing their code, dependencies, and the permissions granted to each function.

## Microservices

An architecture style that breaks an application into small, independent services that communicate over APIs, rather than one large monolithic application.

1. Agility — small, independent services can be updated or changed quickly without redeploying the whole application.
2. Scalability — individual services can be scaled up or down based on demand, rather than scaling the entire application.
3. Faster development — teams can work on different services in parallel.
4. Easy maintenance — issues can often be isolated and fixed within a single service.
5. Improved fault tolerance — a failure in one service doesn't necessarily bring down the whole application.
6. Decomposition — breaking a large application into smaller, focused components.
7. Independence — each service can use its own technology stack and be deployed on its own schedule.

*Security note:* microservices increase the number of APIs and service-to-service communication paths, expanding the attack surface and making authentication/authorization between services (not just to the end user) critical.

## Network Infrastructure

| OSI Layer | Layer Name | Function | Related devices/concepts |
|---|---|---|---|
| 1 | Physical | Transmits raw bits over a physical medium (cabling, radio signal). | Wireless Access Point (WAP) — physical/data-link connectivity for wireless clients. |
| 2 | Data Link | Handles MAC addressing and switching within a local network segment. | Switch — forwards frames based on MAC address; WAP also operates here for wireless frame forwarding. |
| 3 | Network | Handles logical (IP) addressing and routing between different networks. | Router — routes traffic between networks; Layer 3 switch — combines switching with routing; ARP (Address Resolution Protocol) — resolves an IP address (Layer 3) to a MAC address (Layer 2). |
| 4 | Transport | Manages end-to-end connections, segmentation, and reliable delivery. | Load balancer (Layer 4 mode) — distributes traffic based on IP/port without inspecting application content. |
| 7 | Application | Inspects and filters based on actual application content/protocols. | Web Application Firewall (WAF) — filters HTTP traffic for attacks like SQLi/XSS; Network Intrusion Prevention System (NIPS) — inspects traffic across layers and blocks malicious activity; Load balancer (Layer 7 mode) — routes based on application data like URLs or headers. |

## Physical Isolation

Separating a system or network entirely at the physical level, with no shared physical infrastructure (e.g., an air-gapped system with no network connection at all). Provides very strong protection but limits usability and makes updates/monitoring harder.

## Logical Segmentation

1. Subnetting — dividing a network into smaller logical sub-networks using IP addressing.
2. Virtual Local Area Network (VLAN) — logically separating devices on the same physical switch into distinct broadcast domains without separate hardware.

## Software-Defined Networking (SDN)

An approach that separates network control from the physical hardware forwarding traffic, allowing centralized, programmatic management of the network.

1. Management plane — used by administrators to configure and monitor the overall network (e.g., a controller's dashboard/API).
2. Control plane — makes decisions about how traffic should be routed/forwarded, then pushes those decisions down.
3. Data plane — actually forwards traffic based on the rules handed down by the control plane.

## On-Premises

Infrastructure that is owned, hosted, and managed entirely within an organization's own facilities. Gives full control over security and configuration, but also full responsibility for physical security, hardware maintenance, patching, and availability.

## Centralized vs. Decentralized

1. Centralized — management and control are consolidated in one location or system, making it easier to enforce consistent policy but creating a single point of failure if that central point is compromised or goes down.
2. Decentralized — management and control are distributed across multiple locations or nodes, improving resilience (no single point of failure) but making it harder to enforce consistent security policy everywhere.

## Containerization

Packaging an application and its dependencies together into a lightweight, portable "container" that shares the host OS kernel, rather than requiring a full separate OS like a virtual machine. Containers start faster and use fewer resources than VMs, but share the same kernel, so a kernel-level vulnerability can potentially affect all containers on that host.

## Virtualization

Running multiple virtual machines on a single physical host through a hypervisor, each with its own (fully separate) OS. More resource-heavy than containers but provides stronger isolation between workloads.

## IoT (Internet of Things)

1. Lack of standardization — IoT devices come from countless manufacturers with inconsistent security practices and protocols.
2. Data privacy concerns — many IoT devices collect sensitive data (location, health, audio/video) that must be protected.
3. Insecure communication — many devices transmit data without proper encryption.
4. Lifecycle management — IoT devices often stay in use long after they stop receiving security updates.
5. Physical attacks — many IoT devices are deployed in accessible, unmonitored locations, making physical tampering easier.
6. Supply chain risk — devices or their components could be compromised before ever reaching the organization.
7. User awareness — end users often don't realize IoT devices need the same security attention (changing default passwords, updating firmware) as traditional computers.

## ICS/SCADA

Industrial Control Systems / Supervisory Control and Data Acquisition — systems that monitor and control physical industrial processes. Commonly modeled in levels:

1. Plant level — the physical equipment and processes being controlled (motors, valves, sensors).
2. Controller — devices that directly interact with physical equipment based on programmed logic.
3. Coordinating computer level — systems that aggregate data from multiple controllers and coordinate overall process control.
4. Program Logic Controller (PLC) — a ruggedized industrial computer that executes control logic for physical processes in real time.

Common ICS/SCADA sectors:

5. Energy — power generation, transmission, and distribution.
6. Facilities — building management systems (HVAC, elevators, physical access).
7. Manufacturing — production line automation and control.
8. Logistics — automated warehousing, shipping, and supply chain systems.
9. Industrial — broader heavy industry process control (mining, chemical processing, etc.).

*Security note:* these systems were historically designed for reliability and uptime rather than security, and many can't be patched without halting critical physical processes.

## RTOS (Real-Time Operating System)

An operating system designed to process inputs and produce outputs within a strict, guaranteed time constraint — used in systems where timing is critical (medical devices, industrial controllers, automotive systems). Security features are often minimized to preserve the deterministic timing the system depends on, making RTOS-based devices a common weak point.

## Embedded Systems

Purpose-built computing systems built into a larger device to perform a specific function (e.g., a printer's controller, a smart thermostat, a car's engine control unit). Typically have limited processing power/memory, run specialized or proprietary firmware, and are often difficult or impossible to patch.

## High Availability

Designing systems to minimize downtime and keep services running continuously, typically through redundancy — clustering, load balancing, and failover mechanisms that allow another component to take over if one fails.

## Considerations for Your Infrastructure

1. Availability — how consistently the system needs to remain accessible and operational.
2. Resilience — the system's ability to withstand and recover from failures or attacks.
3. Cost — the financial investment required to build, run, and maintain the infrastructure.
4. Responsiveness — how quickly the system reacts to changes in demand or conditions.
5. Scalability — the ability to grow (or shrink) capacity as needs change.
6. Ease of deployment — how quickly and simply new infrastructure or changes can be rolled out.
7. Risk transference — shifting some risk to a third party (e.g., via cloud provider agreements or insurance) rather than holding it entirely in-house.
8. Ease of recovery — how quickly and reliably the system can be restored after a failure or incident.
9. Patch availability — whether the vendor actively provides security updates for the system.
10. Inability to patch — accounting for systems (like some ICS/embedded devices) that can't be patched at all, requiring compensating controls instead.
11. Power — the electrical requirements and redundancy needed to keep infrastructure running.
12. Compute — the processing capacity required to support the workload, now and as it scales.
