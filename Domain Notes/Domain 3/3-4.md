# 3.4 Notes

## High Availability

Minimizing downtime and keeping services running continuously, often through load balancing strategies:

1. Least utilized host — the load balancer sends the next request to whichever server currently has the lightest load.
2. Affinity (session persistence) — ensures a specific client's requests keep going to the same server, preserving session state.
3. DNS round robin — a simple load distribution method where DNS responses cycle through a list of server IP addresses in turn.

## Load Balancing Configurations

1. Active/active — all nodes handle traffic simultaneously, sharing the load and providing both performance and redundancy.
2. Active/passive — only one node handles traffic at a time while the other(s) stay on standby, ready to take over if the active node fails.

## Clustering

Grouping multiple servers together so they act as a single system, providing redundancy and failover.

1. Active/passive node configuration — one node actively serves requests while a standby node waits to take over if the active node fails.
2. Quorum disk — shared storage used by cluster nodes to agree on the current state of the cluster and avoid conflicting actions if communication is disrupted.
3. Witness server — a separate server that helps a cluster determine which node should be active, preventing a "split-brain" scenario where two nodes both think they're in charge.
4. Heartbeat communication — continuous signals exchanged between cluster nodes to confirm each is still alive and functioning; a missed heartbeat can trigger failover.
5. Virtual IP — a single, shared IP address that clients connect to, which the cluster redirects to whichever node is currently active, making failover transparent to users.

## Site Considerations

1. Hot site — a fully operational, ready-to-go backup location with near-instant failover capability; the most expensive but fastest option.
2. Warm site — a partially equipped backup location that needs some setup (restoring recent backups, configuring systems) before it's fully operational.
3. Cold site — a basic location with space and power but little to no pre-installed equipment; the cheapest option but has the longest recovery time.
4. Geographic dispersion — spreading sites/resources across different physical locations so a single regional disaster (natural or otherwise) doesn't take down all of them at once.

## Cloud Data Replication

1. LRS (Locally Redundant Storage) — replicates data multiple times within a single physical location (data center), protecting against hardware failure but not a site-wide disaster.
2. ZRS (Zone Redundant Storage) — replicates data across multiple availability zones within the same region, protecting against a single data center failure.
3. GRS (Geo-Redundant Storage) — replicates data to a second, geographically distant region, protecting against a regional disaster.
4. GZRS (Geo-Zone-Redundant Storage) — combines ZRS within the primary region with GRS-style replication to a secondary region, providing the highest level of durability.

## Data Sovereignty

The principle that data is subject to the laws of the country or region in which it is physically stored or collected. This affects where cloud data can legally be replicated or processed, and organizations must ensure their replication and backup strategy doesn't violate data residency requirements.

## Platform (Resilience Considerations)

1. Redundancy — having duplicate components or systems so a single failure doesn't cause an outage.
2. Adaptability — the ability of the platform to adjust to changing demands or conditions.
3. Resilience against evolving threats — designing systems to withstand new and emerging attack techniques, not just known ones.
4. Enhanced recovery options — having multiple, well-tested ways to restore service after a failure or incident.
5. Compliance and regulation — ensuring the platform's resilience measures also satisfy relevant legal and regulatory requirements.

## Multi-Cloud Systems

Using more than one cloud provider to meet business needs.

1. Resilience against downtime — if one provider has an outage, workloads can potentially continue running on another.
2. Flexibility and choice — organizations can select the best provider or service for each specific need.
3. Cost optimization — comparing and choosing providers based on pricing for specific workloads.
4. Avoiding vendor lock-in — reduces dependency on a single provider's ecosystem, pricing, and terms.

**Challenges:**

1. Complexity — managing multiple providers with different tools, APIs, and configurations increases operational overhead.
2. Security and compliance — maintaining consistent security controls and meeting compliance requirements across different providers is harder.
3. Cost management — tracking and optimizing spend across multiple billing systems and pricing models can be difficult.
4. Integration — making systems across different providers work together smoothly requires additional engineering effort.

## Continuity of Operations

1. Resilience and redundancy — building systems and processes that can keep functioning, or quickly resume, despite disruption.
2. Communication plans — predefined methods and messaging for keeping stakeholders informed during a disruption.
3. Personnel preparedness — ensuring staff know their roles and responsibilities during an incident or disaster.
4. Review and updates — regularly revisiting and updating continuity plans so they stay accurate as the organization changes.

## Capacity Planning

1. Capacity planning for people
   - Skill set assessment — evaluating whether current staff have the skills needed to support future demand.
   - Workload distribution — ensuring work is spread appropriately across available staff.
   - Talent acquisition and development — hiring and training to fill skill or staffing gaps.
   - Succession planning — preparing for the loss of key personnel by developing others who can step into critical roles.
2. Capacity planning for technology
   - Resource scalability — ensuring systems can scale up or down to meet demand.
   - Hardware and software upgrades — planning for necessary upgrades before current systems become a bottleneck or unsupported.
   - Security and compliance — ensuring capacity growth doesn't outpace the organization's ability to secure and govern new resources.
   - Innovation and emerging technologies — evaluating new technologies that could improve capacity or efficiency.
3. Capacity planning for infrastructure
   - Facility expansion and optimization — ensuring physical facilities (data centers, offices) can support growth.
   - Energy efficiency — planning power and cooling needs as infrastructure scales.
   - Disaster recovery — ensuring capacity plans account for the resources needed to recover after a disruption, not just normal operations.

## Testing

1. Tabletop exercise — a discussion-based session where stakeholders walk through a simulated incident scenario without affecting real systems.
2. Failover — testing that a system successfully switches to a backup/redundant component when the primary fails.
3. Simulation — running a more realistic, hands-on test of a scenario to see how systems and people actually respond.
4. Parallel processing — running the backup/recovery system alongside the live production system to validate it works correctly without actually cutting over.

## Backups

1. Full backup — backs up all selected data every time; simplest to restore from, but takes the most time and storage.
2. Incremental — backs up only the data that has changed since the last backup (of any type); fastest to create, but restoration requires the last full backup plus every incremental since.
3. Differential — backs up all data that has changed since the last full backup; faster to restore than incremental (only needs the full backup plus the latest differential), but grows larger over time until the next full backup.

## Important Backup Features

1. On-site vs. off-site
   - On-site — backups stored at the same physical location as the original data; fast to access/restore, but vulnerable to the same site-wide disaster.
   - Off-site — backups stored at a different physical location; protects against site-wide disasters but may be slower to access.
2. Backup frequency — how often backups are taken, balancing data loss tolerance (RPO) against storage/performance cost.
3. Encryption — protecting backup data from unauthorized access, whether at rest or in transit.
4. Snapshots — a point-in-time copy of a system or data set, often used for quick recovery of virtual machines or storage volumes.
5. Recovery — the actual process of restoring data or systems from a backup after loss or failure.
6. Replication — continuously copying data to another location in near real-time, rather than on a scheduled backup basis.
7. Journaling — recording every change made to data in a sequential log, allowing recovery to any specific point in time.

## Power

1. Generators — provide long-term backup power during extended outages, typically fueled by diesel or natural gas.
2. UPS (Uninterruptible Power Supply) — provides short-term battery backup power, enough to bridge a brief outage or allow systems to shut down safely/generators to start.
3. PDU (Power Distribution Unit) — distributes electrical power to multiple devices within a rack or data center, often with monitoring and remote management capabilities.
