# 1.4 Notes

## Public Key Infrastructure (PKI)

1. Public key — shared openly; used to encrypt data or verify a digital signature. Anyone can have a copy of it.
2. Private key — kept secret by its owner; used to decrypt data encrypted with the matching public key or to create a digital signature.
3. Key escrow — a trusted third party holds a copy of private keys, allowing recovery of encrypted data if the original key is lost (or if legally compelled to provide access).

## Encryption

1. Level — encryption can be applied at different layers, from a single record to an entire disk, depending on what needs protecting.
2. Full-disk encryption — encrypts an entire drive automatically, protecting all data at rest if the device is lost or stolen.
3. File encryption — encrypts individual files selectively rather than the whole disk.
4. Volume encryption — encrypts a specific partition or logical volume rather than the whole physical disk.
5. Database encryption — encrypts an entire database, protecting all the data it stores.
6. Record-level encryption — encrypts individual records or fields within a database, allowing more granular protection of sensitive entries.
7. Transport encryption — protects data while it moves across a network, commonly using TLS (the modern replacement for the older, deprecated SSL).

## TLS (Transport Layer Security)

1. Handshake — the initial negotiation process where the client and server agree on protocol version, cipher suite, and exchange keys before secure communication begins.
2. Encryption — scrambling data using an agreed-upon algorithm so it can't be read in transit.
3. Transmission — the actual sending of encrypted data between client and server.
4. Decryption — the receiving party uses the appropriate key to convert the encrypted data back to its original form.
5. Asymmetric — used during the handshake for secure key exchange and authentication (e.g., RSA, ECC), using a public/private key pair.
6. Symmetric — used for the actual bulk data encryption after the handshake (e.g., AES), since it's much faster than asymmetric encryption.
7. Key exchange — the method (e.g., Diffie-Hellman) used to securely establish a shared symmetric key between two parties over an insecure channel.
8. Algorithms — the specific cryptographic methods used for encryption, hashing, and key exchange (e.g., AES, RSA, SHA-256).
9. Key length — the size of the key (e.g., 128-bit, 256-bit) — generally, longer keys are more resistant to brute-force attacks.

## Tools

1. Trusted Platform Module (TPM) — a hardware chip built into a device that securely generates and stores cryptographic keys, supporting features like full-disk encryption and secure boot.
2. Hardware Security Module (HSM) — a dedicated, often removable or networked device used for high-volume key management and cryptographic operations, typically in enterprise or data center environments.
3. Key management system — software/infrastructure used to generate, distribute, store, rotate, and revoke cryptographic keys throughout their lifecycle.
4. Secure enclave — an isolated, protected area of a processor where sensitive data can be processed and keys stored, separate from the main OS.
5. Obfuscation — techniques that disguise or hide the true nature of data:
   - Steganography — hides the existence of a message, often inside an image or file.
   - Tokenization — replaces sensitive data with a non-sensitive placeholder token that has no exploitable value on its own.
   - Data masking — replaces sensitive data with realistic but fake values, often used in non-production environments.
6. Hashing — a one-way function that produces a fixed-length digest of data, used to verify integrity.
7. Salting — adding random data to a value (like a password) before hashing, so identical inputs produce different hashes and defeat rainbow table attacks.
8. Digital signatures — cryptographic proof that verifies the sender's identity and confirms a message wasn't altered.
9. Key stretching — deliberately slowing down a hashing/key derivation function (e.g., PBKDF2, bcrypt, Argon2) to make brute-force attacks more computationally expensive.
10. Blockchain — a distributed, cryptographically linked ledger of records (blocks) where each block references the previous one, making tampering evident.
11. Open public ledger — a blockchain-based ledger that anyone can view and verify, providing transparency without needing a central trusted authority.

## Certification

1. Certificate Authorities (CAs) — trusted entities that issue and sign digital certificates, binding a public key to a verified identity.
2. Certificate Revocation List (CRL) — a list published by a CA of certificates that have been revoked before their expiration date.
3. Online Certificate Status Protocol (OCSP) — a real-time protocol used to check whether a specific certificate has been revoked, without needing to download a full CRL.
4. Self-signed — a certificate signed by the same entity that created it, rather than by a trusted CA; useful for internal testing but not trusted by default by browsers/clients.
5. Third-party — a certificate issued by an external, trusted CA rather than generated internally.
6. Root of trust — the foundational, implicitly trusted certificate (root CA) from which the entire chain of trust is built.
7. Certificate Signing Request (CSR) generation — the process of creating a request (containing the public key and identifying information) that is sent to a CA to obtain a signed certificate.
8. Wildcard — a certificate that secures a domain and all of its first-level subdomains (e.g., *.example.com) with a single certificate.
