# 2.5 Notes

## Segmentation

1. Physical segmentation — separating networks using distinct physical hardware (separate switches, cabling, routers) so there's no shared infrastructure between zones.
2. Virtual Local Area Networks (VLANs) — logically separating devices on the same physical switch into distinct broadcast domains, without needing separate hardware.
3. Subnetting — dividing a network into smaller logical sub-networks using IP addressing, to organize and control traffic flow.
4. Micro-segmentation — granular, often software-defined isolation down to individual workloads or applications, going beyond traditional network segmentation.
5. Enhanced security — segmentation limits how far an attacker can move if one segment is compromised, containing the blast radius of a breach.
6. Access control — segmentation supports enforcing which users/systems can reach which resources, adding another layer beyond basic network rules.
7. Compliance requirements — segmentation is often required to meet regulatory standards, such as PCI DSS (Payment Card Industry Data Security Standard) for cardholder data, or HIPAA (Health Insurance Portability and Accountability Act) for PHI (Protected Health Information).
8. Performance optimization — smaller, segmented broadcast domains reduce unnecessary traffic and congestion, improving network performance.
9. Isolation of critical systems — high-value systems (e.g., domain controllers, ICS/SCADA) can be placed in their own segment, separated from general user traffic.
10. Scalability and agility — segmented networks are easier to expand, reconfigure, or troubleshoot without impacting the entire environment.

## Access Control List (ACL)

A set of rules configured on a network device (router, switch, firewall) that explicitly permits or denies traffic based on criteria like source/destination IP, port, or protocol. ACLs are evaluated in order, and most devices apply an implicit "deny all" at the end of the list for anything that doesn't match a rule.

## Application Allow List

A list of explicitly approved applications that are permitted to run; anything not on the list is blocked by default. This is a restrictive, "default-deny" approach that's more secure but requires more maintenance.

## Application Block List

A list of explicitly prohibited applications that are blocked from running; everything else is allowed by default. This is a "default-allow" approach — easier to maintain than an allow list, but less secure since new/unknown threats aren't blocked automatically.

## Isolation

Separating a system, application, or network segment from the rest of the environment so that if it's compromised, the damage can't spread. Common examples include sandboxing a suspicious file, quarantining an infected host, or air-gapping a highly sensitive system entirely.

## Patching

Applying vendor-released updates to fix known vulnerabilities, bugs, or security flaws in software, firmware, and operating systems. Patch management typically involves testing patches before deployment, prioritizing based on severity (e.g., CVSS score), and applying them on a regular, documented schedule.

## Encryption

Protecting the confidentiality of data by converting it into an unreadable format that can only be reversed with the correct key — applied to data at rest, in transit, and (less commonly) in use, as a mitigation against interception or theft.

## Monitoring

1. SIEM (Security Information and Event Management) — aggregates and correlates logs from across the environment to detect and alert on threats.
2. SOAR (Security Orchestration, Automation, and Response) — builds on SIEM data by automating and orchestrating response playbooks, reducing manual effort during incident response.

## Least Privilege

The principle that users, applications, and systems should be granted only the minimum level of access necessary to perform their function — nothing more. This limits the potential damage from a compromised account or process.

## Configuration Enforcement

1. Standardization — applying consistent, approved configurations across all systems to reduce unexpected gaps or weaknesses.
2. Vulnerability mitigation — enforced baseline configurations close off unnecessary settings or services that could otherwise be exploited.
3. Compliance adherence — enforced configurations help ensure systems consistently meet regulatory or internal policy requirements.
4. Automation — using tools to automatically apply, check, and correct configurations at scale, rather than relying on manual, error-prone processes.

## Decommissioning

The formal process of retiring a system or device from use, ensuring it no longer poses a security risk after it's taken offline.

1. Documentation — recording what was decommissioned, when, and by whom, to maintain an accurate asset inventory and audit trail.
2. Data sanitization — ensuring no recoverable sensitive data remains on the device before disposal or reuse:
   - Paper sanitization — securely destroying physical documents (e.g., shredding, pulping, incineration).
   - Media sanitization — securely erasing or destroying digital storage media (e.g., degaussing, wiping, physical destruction/shredding of drives).

## Hardening Techniques

1. EDR (Endpoint Detection and Response) — continuously monitors endpoint behavior to detect and respond to threats beyond what signature-based antivirus can catch.
2. Continuous monitoring — ongoing, real-time observation of systems and traffic rather than periodic checks, to catch issues as they happen.
3. Behavioral analysis — identifying threats by detecting abnormal patterns of activity rather than relying solely on known signatures.
4. Threat detection — the overall process of identifying malicious activity through monitoring, analysis, and alerting tools.
5. Alert generation — automatically notifying security staff when suspicious activity is detected, so it can be investigated quickly.
6. Response and remediation — taking action to contain, remove, and recover from a detected threat.
7. Forensic analysis — investigating an incident after the fact to determine root cause, scope, and impact, often supporting legal or compliance needs.
8. Real-time threat mitigation — automatically blocking or containing a threat as it's detected, without waiting for manual intervention.
9. Improved incident response — hardening tools like EDR give responders better data and faster action, shortening the time from detection to resolution.
10. Endpoint visibility — clear insight into what's happening on every endpoint, which is essential for detecting subtle or novel attacks.
11. Host-based Intrusion Prevention System (HIPS) — software running on an individual host that monitors and actively blocks malicious activity on that specific device.
12. Host-based firewall — a firewall running locally on a single device, filtering traffic to and from that specific host rather than an entire network.
13. Disabling unnecessary ports — closing ports that aren't needed reduces the attack surface. Common ports to review:

   | Port | Protocol/Service | Why it matters |
   |---|---|---|
   | 21 | FTP | Unencrypted file transfer; disable if not needed, or replace with SFTP (22). |
   | 23 | Telnet | Unencrypted remote access; should be disabled in favor of SSH (22). |
   | 25 | SMTP | Mail relay; open only where an actual mail server needs it. |
   | 135/137-139 | Windows RPC/NetBIOS | Common target for lateral movement on internal networks; restrict to trusted hosts. |
   | 445 | SMB | File sharing; frequently exploited (e.g., worms); restrict/disable on internet-facing systems. |
   | 3389 | RDP | Remote desktop; a top target for brute-force and ransomware attacks if exposed to the internet. |

14. Default password change — changing factory-default credentials on devices and software immediately after deployment, since default credentials are widely known and often published online.
15. Removal of unnecessary software — uninstalling applications, services, or features that aren't required, reducing the number of potential vulnerabilities on a system.
